# Sanity tests

Database should be validated against sanity checks. Each sanity check should make sure that data is sound and correctly linked to other data sources.

For example:
 * each person should have one unique ID
 * each parliamentary session has a key and a date
 * each parliamentary section has one session
 * each section has a type, title and description
 * number of people in parliament NEVER exceeds 200
   * it should be 200 in most cases, whenever this is not a case this is an esception, usually there is a reason (e.g. member left, no one joined yet)
 * number of votes in a voting NEVER exceeds 199 (if exceeded, this is unique exception and even then should have exactly 200)
 * each hallituksen esitys (HE) should be able to have resolution if not open
 * votings should be linked to single section (i.e.) budget votings should not cross parliamentary boundaries

 Consult the domain expert agent for additional requirements.

 The sanity checks should be executed against _final_ database, and output stored in a way that it can be read by the server to be output to the client.
